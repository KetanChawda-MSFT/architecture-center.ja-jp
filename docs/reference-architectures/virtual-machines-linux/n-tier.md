<span data-ttu-id="430f6-180">ラック認識モードでノードを構成します。</span><span class="sxs-lookup"><span data-stu-id="430f6-180">Configure nodes in rack-aware mode.</span></span> <span data-ttu-id="430f6-181">`cassandra-rackdc.properties` ファイル内で障害ドメインをラックにマッピングします。</span><span class="sxs-lookup"><span data-stu-id="430f6-181">Map fault domains to racks in the `cassandra-rackdc.properties` file.</span></span>

<span data-ttu-id="430f6-182">クラスターの前にロード バランサーは必要ありません。</span><span class="sxs-lookup"><span data-stu-id="430f6-182">You don't need a load balancer in front of the cluster.</span></span> <span data-ttu-id="430f6-183">クライアントはクラスターのノードに直接接続します。</span><span class="sxs-lookup"><span data-stu-id="430f6-183">The client connects directly to a node in the cluster.</span></span>

### <a name="jumpbox"></a><span data-ttu-id="430f6-184">Jumpbox</span><span class="sxs-lookup"><span data-stu-id="430f6-184">Jumpbox</span></span>

<span data-ttu-id="430f6-185">ジャンプボックスのパフォーマンス要件は最小限に抑えられるため、ジャンプボックスには Standard A1 などの小さな VM のサイズを選択します。</span><span class="sxs-lookup"><span data-stu-id="430f6-185">The jumpbox will have minimal performance requirements, so select a small VM size for the jumpbox such as Standard A1.</span></span> 

<span data-ttu-id="430f6-186">ジャンプボックス用に[パブリック IP アドレス]を作成します。</span><span class="sxs-lookup"><span data-stu-id="430f6-186">Create a [public IP address] for the jumpbox.</span></span> <span data-ttu-id="430f6-187">ジャンプボックスを、他の VM と同じ VNet 内の、個別の管理サブネット内に配置します。</span><span class="sxs-lookup"><span data-stu-id="430f6-187">Place the jumpbox in the same VNet as the other VMs, but in a separate management subnet.</span></span>

<span data-ttu-id="430f6-188">アプリケーション ワークロードを実行する VM へのパブリック インターネットからの SSH アクセスを許可しないでください。</span><span class="sxs-lookup"><span data-stu-id="430f6-188">Do not allow SSH access from the public Internet to the VMs that run the application workload.</span></span> <span data-ttu-id="430f6-189">代わりに、これらの VM へのすべての SSH アクセスは、ジャンプボックスを経由する必要があります。</span><span class="sxs-lookup"><span data-stu-id="430f6-189">Instead, all SSH access to these VMs must come through the jumpbox.</span></span> <span data-ttu-id="430f6-190">管理者はジャンプボックスにログインし、次にジャンプボックスから他の VM にログインします。</span><span class="sxs-lookup"><span data-stu-id="430f6-190">An administrator logs into the jumpbox, and then logs into the other VM from the jumpbox.</span></span> <span data-ttu-id="430f6-191">ジャンプボックスは、既知の安全な IP アドレスからのみ、インターネットからの SSH トラフィックを許可します。</span><span class="sxs-lookup"><span data-stu-id="430f6-191">The jumpbox allows SSH traffic from the Internet, but only from known, safe IP addresses.</span></span>

<span data-ttu-id="430f6-192">ジャンプボックスをセキュリティで保護するには、NSG を作成してジャンプボックスのサブネットに適用します。</span><span class="sxs-lookup"><span data-stu-id="430f6-192">To secure the jumpbox, create an NSG and apply it to the jumpbox subnet.</span></span> <span data-ttu-id="430f6-193">安全な一連のパブリック IP アドレスからのみ SSH 接続を許可する NSG ルールを追加します。</span><span class="sxs-lookup"><span data-stu-id="430f6-193">Add an NSG rule that allows SSH connections only from a safe set of public IP addresses.</span></span> <span data-ttu-id="430f6-194">NSG は、サブネットまたはジャンプボックスの NIC のいずれかに関連付けることができます。</span><span class="sxs-lookup"><span data-stu-id="430f6-194">The NSG can be attached either to the subnet or to the jumpbox NIC.</span></span> <span data-ttu-id="430f6-195">この場合は NIC に関連付けることをお勧めします。こうすることで、仮に同じサブネットに別の VM を追加した場合でも、SSH トラフィックはジャンプボックスのみに許可されます。</span><span class="sxs-lookup"><span data-stu-id="430f6-195">In this case, we recommend attaching it to the NIC, so SSH traffic is permitted only to the jumpbox, even if you add other VMs to the same subnet.</span></span>

<span data-ttu-id="430f6-196">他のサブネットに対しても NSG を構成して、管理サブネットからの SSH トラフィックを許可します。</span><span class="sxs-lookup"><span data-stu-id="430f6-196">Configure the NSGs for the other subnets to allow SSH traffic from the management subnet.</span></span>

## <a name="availability-considerations"></a><span data-ttu-id="430f6-197">可用性に関する考慮事項</span><span class="sxs-lookup"><span data-stu-id="430f6-197">Availability considerations</span></span>

<span data-ttu-id="430f6-198">各層または各 VM ロールを、個別の可用性セットに配置します。</span><span class="sxs-lookup"><span data-stu-id="430f6-198">Put each tier or VM role into a separate availability set.</span></span> 

<span data-ttu-id="430f6-199">データベース層に複数の VM を備えても、自動的にデータベースの可用性が高くなるわけではありません。</span><span class="sxs-lookup"><span data-stu-id="430f6-199">At the database tier, having multiple VMs does not automatically translate into a highly available database.</span></span> <span data-ttu-id="430f6-200">リレーショナル データベースの場合は、通常、高可用性を実現するにはレプリケーションとフェールオーバーを使用する必要があります。</span><span class="sxs-lookup"><span data-stu-id="430f6-200">For a relational database, you will typically need to use replication and failover to achieve high availability.</span></span>  

<span data-ttu-id="430f6-201">[VM の Azure SLA][vm-sla] によって提供されるものよりも高い可用性が必要な場合は、2 つのリージョンにアプリケーションをレプリケートし、Azure Traffic Manager を使用してフェールオーバーを行います。</span><span class="sxs-lookup"><span data-stu-id="430f6-201">If you need higher availability than the [Azure SLA for VMs][vm-sla] provides, replicate the application across two regions and use Azure Traffic Manager for failover.</span></span> <span data-ttu-id="430f6-202">詳細については、「[高可用性を得るために複数のリージョンで Linux VM を実行する][multi-dc]」を参照してください。</span><span class="sxs-lookup"><span data-stu-id="430f6-202">For more information, see [Run Linux VMs in multiple regions for high availability][multi-dc].</span></span>  

## <a name="security-considerations"></a><span data-ttu-id="430f6-203">セキュリティに関する考慮事項</span><span class="sxs-lookup"><span data-stu-id="430f6-203">Security considerations</span></span>

## <a name="scalability-considerations"></a><span data-ttu-id="430f6-207">拡張性に関する考慮事項</span><span class="sxs-lookup"><span data-stu-id="430f6-207">Scalability considerations</span></span>
