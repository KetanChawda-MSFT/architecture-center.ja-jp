# <a name="designing-microservices-interservice-communication"></a><span data-ttu-id="b724e-103">マイクロサービスの設計:サービス間の通信</span><span class="sxs-lookup"><span data-stu-id="b724e-103">Designing microservices: Interservice communication</span></span>

## <a name="challenges"></a><span data-ttu-id="b724e-109">課題</span><span class="sxs-lookup"><span data-stu-id="b724e-109">Challenges</span></span>

<span data-ttu-id="b724e-110">ここでは、サービス間通信において発生する主要な課題を示します。</span><span class="sxs-lookup"><span data-stu-id="b724e-110">Here are some of the main challenges arising from service-to-service communication.</span></span> <span data-ttu-id="b724e-111">サービス メッシュ (後で説明します) は、これらの課題の多くに対処するように設計されています。</span><span class="sxs-lookup"><span data-stu-id="b724e-111">Service meshes, described later in this chapter, are designed to handle many of these challenges.</span></span>

<span data-ttu-id="b724e-112">**回復性**。</span><span class="sxs-lookup"><span data-stu-id="b724e-112">**Resiliency**.</span></span> <span data-ttu-id="b724e-113">特定のマイクロサービスには、数十から場合によっては数百ものインスタンスが存在する可能性があります。</span><span class="sxs-lookup"><span data-stu-id="b724e-113">There may be dozens or even hundreds of instances of any given microservice.</span></span> <span data-ttu-id="b724e-114">インスタンスは、さまざまな理由で失敗することがあります。</span><span class="sxs-lookup"><span data-stu-id="b724e-114">An instance can fail for any number of reasons.</span></span> <span data-ttu-id="b724e-115">ハードウェア障害や仮想マシンの再起動のように、ノード レベルで障害が発生する場合があります。</span><span class="sxs-lookup"><span data-stu-id="b724e-115">There can be a node-level failure, such as a hardware failure or a VM reboot.</span></span> <span data-ttu-id="b724e-116">インスタンスがクラッシュしたり、要求の負荷が大きすぎて新しい要求を処理できなくなる場合もあります。</span><span class="sxs-lookup"><span data-stu-id="b724e-116">An instance might crash, or be overwhelmed with requests and unable to process any new requests.</span></span> <span data-ttu-id="b724e-117">これらのどのイベントによっても、ネットワーク呼び出しが失敗する可能性があります。</span><span class="sxs-lookup"><span data-stu-id="b724e-117">Any of these events can cause a network call to fail.</span></span> <span data-ttu-id="b724e-118">サービス間ネットワーク呼び出しの回復力を高めるのに役立つ 2 つの設計パターンがあります。</span><span class="sxs-lookup"><span data-stu-id="b724e-118">There are two design patterns that can help make service-to-service network calls more resilient:</span></span>

- <span data-ttu-id="b724e-119">**[再試行](../patterns/retry.md)**。</span><span class="sxs-lookup"><span data-stu-id="b724e-119">**[Retry](../patterns/retry.md)**.</span></span> <span data-ttu-id="b724e-120">何もしなくても自動的に解消される一時的なエラーのために、ネットワーク呼び出し失敗することがあります。</span><span class="sxs-lookup"><span data-stu-id="b724e-120">A network call may fail because of a transient fault that goes away by itself.</span></span> <span data-ttu-id="b724e-121">呼び出し元は、通常、すぐに失敗として処理するのではなく、一定の回数だけ、または構成されたタイムアウト期間が経過するまで、操作を再試行する必要があります。</span><span class="sxs-lookup"><span data-stu-id="b724e-121">Rather than fail outright, the caller should typically retry the operation a certain number of times, or until a configured time-out period elapses.</span></span> <span data-ttu-id="b724e-122">ただし、操作がべき等でない場合は、再試行によって意図しない副作用が発生する可能性があります。</span><span class="sxs-lookup"><span data-stu-id="b724e-122">However, if an operation is not idempotent, retries can cause unintended side effects.</span></span> <span data-ttu-id="b724e-123">その場合、元の呼び出しが成功したとしても、呼び出し元は応答を取得できません。</span><span class="sxs-lookup"><span data-stu-id="b724e-123">The original call might succeed, but the caller never gets a response.</span></span> <span data-ttu-id="b724e-124">呼び出し元が再試行を行うと、操作が 2 回呼び出される可能性があります。</span><span class="sxs-lookup"><span data-stu-id="b724e-124">If the caller retries, the operation may be invoked twice.</span></span> <span data-ttu-id="b724e-125">一般に、POST または PATCH メソッドは、べき等が保証されていないため、再試行することは安全ではありません。</span><span class="sxs-lookup"><span data-stu-id="b724e-125">Generally, it's not safe to retry POST or PATCH methods, because these are not guaranteed to be idempotent.</span></span>

- <span data-ttu-id="b724e-126">**[サーキット ブレーカー](../patterns/circuit-breaker.md)**。</span><span class="sxs-lookup"><span data-stu-id="b724e-126">**[Circuit Breaker](../patterns/circuit-breaker.md)**.</span></span> <span data-ttu-id="b724e-127">失敗した要求が多すぎると、保留中の要求がキューに蓄積されるため、ボトルネックの原因になる可能性があります。</span><span class="sxs-lookup"><span data-stu-id="b724e-127">Too many failed requests can cause a bottleneck, as pending requests accumulate in the queue.</span></span> <span data-ttu-id="b724e-128">これらのブロックされた要求が重要なシステム リソース (メモリ、しきい値、データベース接続など) を保持することで、障害が連鎖する可能性があります。</span><span class="sxs-lookup"><span data-stu-id="b724e-128">These blocked requests might hold critical system resources such as memory, threads, database connections, and so on, which can cause cascading failures.</span></span> <span data-ttu-id="b724e-129">サーキット ブレーカー パターンを使うと、失敗する可能性がある操作をサービスで繰り返し再試行することを回避できます。</span><span class="sxs-lookup"><span data-stu-id="b724e-129">The Circuit Breaker pattern can prevent a service from repeatedly trying an operation that is likely to fail.</span></span>

<span data-ttu-id="b724e-130">**負荷分散**。</span><span class="sxs-lookup"><span data-stu-id="b724e-130">**Load balancing**.</span></span> <span data-ttu-id="b724e-131">サービス "A" がサービス "B" を呼び出すとき、要求はサービス "B" の実行中のインスタンスに到達する必要があります。</span><span class="sxs-lookup"><span data-stu-id="b724e-131">When service "A" calls service "B", the request must reach a running instance of service "B".</span></span> <span data-ttu-id="b724e-132">Kubernetes では、`Service` リソースの種類はポッドのグループに対して安定した IP アドレスを提供します。</span><span class="sxs-lookup"><span data-stu-id="b724e-132">In Kubernetes, the `Service` resource type provides a stable IP address for a group of pods.</span></span> <span data-ttu-id="b724e-133">サービスの IP アドレスへのネットワーク トラフィックは、iptable の規則によってポッドに転送されます。</span><span class="sxs-lookup"><span data-stu-id="b724e-133">Network traffic to the service's IP address gets forwarded to a pod by means of iptable rules.</span></span> <span data-ttu-id="b724e-134">既定では、ポッドはランダムに選ばれます。</span><span class="sxs-lookup"><span data-stu-id="b724e-134">By default, a random pod is chosen.</span></span> <span data-ttu-id="b724e-135">サービス メッシュ (後述) は、観察された待機時間や他のメトリックに基づいて、さらにインテリジェントな負荷分散アルゴリズムを提供できます。</span><span class="sxs-lookup"><span data-stu-id="b724e-135">A service mesh (see below) can provide more intelligent load balancing algorithms based on observed latency or other metrics.</span></span>

<span data-ttu-id="b724e-136">**分散トレース**。</span><span class="sxs-lookup"><span data-stu-id="b724e-136">**Distributed tracing**.</span></span> <span data-ttu-id="b724e-137">1 つのトランザクションで複数のサービスが利用される場合があります。</span><span class="sxs-lookup"><span data-stu-id="b724e-137">A single transaction may span multiple services.</span></span> <span data-ttu-id="b724e-138">その場合、システムの全体的なパフォーマンスと正常性の監視が困難になります。</span><span class="sxs-lookup"><span data-stu-id="b724e-138">That can make it hard to monitor the overall performance and health of the system.</span></span> <span data-ttu-id="b724e-139">すべてのサービスがログとメトリックを生成する場合でも、それらを結び付ける何らかの手段がないと、役に立ちません。</span><span class="sxs-lookup"><span data-stu-id="b724e-139">Even if every service generates logs and metrics, without some way to tie them together, they are of limited use.</span></span> <span data-ttu-id="b724e-140">分散トレースについては「[ログ記録と監視](./logging-monitoring.md)」の章で詳しく説明されていますが、ここでも課題としてそのことに触れます。</span><span class="sxs-lookup"><span data-stu-id="b724e-140">The chapter [Logging and monitoring](./logging-monitoring.md) talks more about distributed tracing, but we mention it here as a challenge.</span></span>

<span data-ttu-id="b724e-141">**サービスのバージョン管理**。</span><span class="sxs-lookup"><span data-stu-id="b724e-141">**Service versioning**.</span></span> <span data-ttu-id="b724e-142">チームは、新しいバージョンのサービスをデプロイするときに、他のサービスや、そのサービスに依存する外部クライアントで、中断が発生しないようにする必要があります。</span><span class="sxs-lookup"><span data-stu-id="b724e-142">When a team deploys a new version of a service, they must avoid breaking any other services or external clients that depend on it.</span></span> <span data-ttu-id="b724e-143">さらに、サービスの複数のバージョンをサイド バイ サイドで実行し、要求を特定のバージョンにルーティングすることが必要になる場合があります。</span><span class="sxs-lookup"><span data-stu-id="b724e-143">In addition, you might want to run multiple versions of a service side-by-side, and route requests to a particular version.</span></span> <span data-ttu-id="b724e-144">この問題について詳しくは、「[API のバージョン管理](./api-design.md#api-versioning)」をご覧ください。</span><span class="sxs-lookup"><span data-stu-id="b724e-144">See [API Versioning](./api-design.md#api-versioning) for more discussion of this issue.</span></span>

<span data-ttu-id="b724e-145">**TLS 暗号化と相互 TLS 認証**。</span><span class="sxs-lookup"><span data-stu-id="b724e-145">**TLS encryption and mutual TLS authentication**.</span></span> <span data-ttu-id="b724e-146">セキュリティ上の理由から、サービス間のトラフィックを TLS で暗号化し、相互 TLS 認証を使って呼び出し元を認証することが必要な場合があります。</span><span class="sxs-lookup"><span data-stu-id="b724e-146">For security reasons, you may want to encrypt traffic between services with TLS, and use mutual TLS authentication to authenticate callers.</span></span>

## <a name="synchronous-versus-asynchronous-messaging"></a><span data-ttu-id="b724e-147">同期メッセージングと非同期メッセージング</span><span class="sxs-lookup"><span data-stu-id="b724e-147">Synchronous versus asynchronous messaging</span></span>

<span data-ttu-id="b724e-148">マイクロサービス間の通信には、次の 2 つの基本的なメッセージング パターンを使うことができます。</span><span class="sxs-lookup"><span data-stu-id="b724e-148">There are two basic messaging patterns that microservices can use to communicate with other microservices.</span></span>

1. <span data-ttu-id="b724e-149">同期通信。</span><span class="sxs-lookup"><span data-stu-id="b724e-149">Synchronous communication.</span></span> <span data-ttu-id="b724e-150">このパターンでは、サービスは、別のサービスによって公開されている API を、HTTP や gRPC などのプロトコルを使って呼び出します。</span><span class="sxs-lookup"><span data-stu-id="b724e-150">In this pattern, a service calls an API that another service exposes, using a protocol such as HTTP or gRPC.</span></span> <span data-ttu-id="b724e-151">このオプションは、呼び出し元が受信側からの応答を待機するため、同期メッセージング パターンです。</span><span class="sxs-lookup"><span data-stu-id="b724e-151">This option is a synchronous messaging pattern because the caller waits for a response from the receiver.</span></span>

2. <span data-ttu-id="b724e-152">非同期のメッセージ受け渡し。</span><span class="sxs-lookup"><span data-stu-id="b724e-152">Asynchronous message passing.</span></span> <span data-ttu-id="b724e-153">このパターンでは、サービスは応答を待たずにメッセージを送信し、1 つまたは複数のサービスがメッセージを非同期的に処理します。</span><span class="sxs-lookup"><span data-stu-id="b724e-153">In this pattern, a service sends message without waiting for a response, and one or more services process the message asynchronously.</span></span>

<span data-ttu-id="b724e-154">非同期 I/O と非同期プロトコルを区別することが重要です。</span><span class="sxs-lookup"><span data-stu-id="b724e-154">It's important to distinguish between asynchronous I/O and an asynchronous protocol.</span></span> <span data-ttu-id="b724e-155">非同期 I/O とは、I/O が実行されている間、呼び出し元のスレッドがブロックされないことを意味します。</span><span class="sxs-lookup"><span data-stu-id="b724e-155">Asynchronous I/O means the calling thread is not blocked while the I/O completes.</span></span> <span data-ttu-id="b724e-156">非同期 I/O はパフォーマンスにとって重要ですが、実装の詳細はアーキテクチャに関係します。</span><span class="sxs-lookup"><span data-stu-id="b724e-156">That's important for performance, but is an implementation detail in terms of the architecture.</span></span> <span data-ttu-id="b724e-157">非同期プロトコルとは、送信側が応答を待機しないことを意味します。</span><span class="sxs-lookup"><span data-stu-id="b724e-157">An asynchronous protocol means the sender doesn't wait for a response.</span></span> <span data-ttu-id="b724e-158">HTTP クライアントは要求を送信するときに非同期 I/O を使うことができたとしても、HTTP は同期プロトコルです。</span><span class="sxs-lookup"><span data-stu-id="b724e-158">HTTP is a synchronous protocol, even though an HTTP client may use asynchronous I/O when it sends a request.</span></span>

<span data-ttu-id="b724e-159">各パターンにはトレードオフがあります。</span><span class="sxs-lookup"><span data-stu-id="b724e-159">There are tradeoffs to each pattern.</span></span> <span data-ttu-id="b724e-160">要求/応答はよく知られたパラダイムであるため、API の設計の方が、メッセージング システムの設計より自然に感じられるかもしれません。</span><span class="sxs-lookup"><span data-stu-id="b724e-160">Request/response is a well-understood paradigm, so designing an API may feel more natural than designing a messaging system.</span></span> <span data-ttu-id="b724e-161">しかし、非同期メッセージングには、マイクロサービス アーキテクチャで非常に役に立つ利点がいくつかあります。</span><span class="sxs-lookup"><span data-stu-id="b724e-161">However, asynchronous messaging has some advantages that can be very useful in a microservices architecture:</span></span>

- <span data-ttu-id="b724e-162">**低い結合性**。</span><span class="sxs-lookup"><span data-stu-id="b724e-162">**Reduced coupling**.</span></span> <span data-ttu-id="b724e-163">メッセージの送信側は、コンシューマーについて知っている必要はありません。</span><span class="sxs-lookup"><span data-stu-id="b724e-163">The message sender does not need to know about the consumer.</span></span>

- <span data-ttu-id="b724e-164">**複数のサブスクライバー**。</span><span class="sxs-lookup"><span data-stu-id="b724e-164">**Multiple subscribers**.</span></span> <span data-ttu-id="b724e-165">パブリッシュ/サブスクライブ モデルを使うと、複数のコンシューマーがイベントの受信をサブスクライブできます。</span><span class="sxs-lookup"><span data-stu-id="b724e-165">Using a pub/sub model, multiple consumers can subscribe to receive events.</span></span> <span data-ttu-id="b724e-166">「[イベント ドリブン アーキテクチャのスタイル](/azure/architecture/guide/architecture-styles/event-driven)」をご覧ください。</span><span class="sxs-lookup"><span data-stu-id="b724e-166">See [Event-driven architecture style](/azure/architecture/guide/architecture-styles/event-driven).</span></span>

- <span data-ttu-id="b724e-167">**障害の分離**。</span><span class="sxs-lookup"><span data-stu-id="b724e-167">**Failure isolation**.</span></span> <span data-ttu-id="b724e-168">コンシューマーで障害が発生した場合でも、送信側はメッセージを送信できます。</span><span class="sxs-lookup"><span data-stu-id="b724e-168">If the consumer fails, the sender can still send messages.</span></span> <span data-ttu-id="b724e-169">メッセージは、コンシューマーが復旧すると取得されます。</span><span class="sxs-lookup"><span data-stu-id="b724e-169">The messages will be picked up when the consumer recovers.</span></span> <span data-ttu-id="b724e-170">各サービスが独自のライフサイクルを持つマイクロサービス アーキテクチャでは、この機能が特に役に立ちます。</span><span class="sxs-lookup"><span data-stu-id="b724e-170">This ability is especially useful in a microservices architecture, because each service has its own lifecycle.</span></span> <span data-ttu-id="b724e-171">サービスがいつ使用不能になっても、またはサービスをいつ新しいバージョンに置き換えても、問題はありません。</span><span class="sxs-lookup"><span data-stu-id="b724e-171">A service could become unavailable or be replaced with a newer version at any given time.</span></span> <span data-ttu-id="b724e-172">非同期メッセージングは、断続的なダウンタイムを処理できます。</span><span class="sxs-lookup"><span data-stu-id="b724e-172">Asynchronous messaging can handle intermittent downtime.</span></span> <span data-ttu-id="b724e-173">一方、同期 API では、ダウンストリーム サービスが使用可能である必要があり、使用できないと操作は失敗します。</span><span class="sxs-lookup"><span data-stu-id="b724e-173">Synchronous APIs, on the other hand, require the downstream service to be available or the operation fails.</span></span>

- <span data-ttu-id="b724e-174">**応答性**。</span><span class="sxs-lookup"><span data-stu-id="b724e-174">**Responsiveness**.</span></span> <span data-ttu-id="b724e-175">アップストリームのサービスは、ダウンストリームのサービスを待つ必要がなければ、より速く応答できます。</span><span class="sxs-lookup"><span data-stu-id="b724e-175">An upstream service can reply faster if it does not wait on downstream services.</span></span> <span data-ttu-id="b724e-176">これは、マイクロサービス アーキテクチャで特に役に立ちます。</span><span class="sxs-lookup"><span data-stu-id="b724e-176">This is especially useful in a microservices architecture.</span></span> <span data-ttu-id="b724e-177">サービスの依存関係が連鎖している場合 (たとえば、サービス A が B を呼び出し、B が C を呼び出している場合)、同期呼び出しでの待機により待機時間が許容できないほど長くなる可能性があります。</span><span class="sxs-lookup"><span data-stu-id="b724e-177">If there is a chain of service dependencies (service A calls B, which calls C, and so on), waiting on synchronous calls can add unacceptable amounts of latency.</span></span>

- <span data-ttu-id="b724e-178">**負荷平準化**。</span><span class="sxs-lookup"><span data-stu-id="b724e-178">**Load leveling**.</span></span> <span data-ttu-id="b724e-179">キューは、受信側が独自の速度でメッセージを処理できるように、ワークロードを平準化するためのバッファーとして機能できます。</span><span class="sxs-lookup"><span data-stu-id="b724e-179">A queue can act as a buffer to level the workload, so that receivers can process messages at their own rate.</span></span>

- <span data-ttu-id="b724e-180">**ワークフロー**。</span><span class="sxs-lookup"><span data-stu-id="b724e-180">**Workflows**.</span></span> <span data-ttu-id="b724e-181">キューを使うと、ワークフローの各ステップの後にメッセージのチェックポイントを設けることで、ワークフローを管理できます。</span><span class="sxs-lookup"><span data-stu-id="b724e-181">Queues can be used to manage a workflow, by check-pointing the message after each step in the workflow.</span></span>

<span data-ttu-id="b724e-182">ただし、非同期メッセージングを効果的に使うためには、いくつかの課題もあります。</span><span class="sxs-lookup"><span data-stu-id="b724e-182">However, there are also some challenges to using asynchronous messaging effectively.</span></span>

- <span data-ttu-id="b724e-188">**コスト**。</span><span class="sxs-lookup"><span data-stu-id="b724e-188">**Cost**.</span></span> <span data-ttu-id="b724e-189">スループットを高くすると、メッセージング インフラストラクチャの金銭的コストが大きくなることがあります。</span><span class="sxs-lookup"><span data-stu-id="b724e-189">At high throughputs, the monetary cost of the messaging infrastructure could be significant.</span></span>

- <span data-ttu-id="b724e-190">**複雑さ**。</span><span class="sxs-lookup"><span data-stu-id="b724e-190">**Complexity**.</span></span> <span data-ttu-id="b724e-191">非同期メッセージングの処理は、簡単なタスクではありません。</span><span class="sxs-lookup"><span data-stu-id="b724e-191">Handling asynchronous messaging is not a trivial task.</span></span> <span data-ttu-id="b724e-192">たとえば、メッセージの重複は、重複除去を行うか、または操作をべき等にすることによって、処理する必要があります。</span><span class="sxs-lookup"><span data-stu-id="b724e-192">For example, you must handle duplicated messages, either by de-duplicating or by making operations idempotent.</span></span> <span data-ttu-id="b724e-193">また、非同期メッセージングを使って要求/応答のセマンティクスを実装するのは困難です。</span><span class="sxs-lookup"><span data-stu-id="b724e-193">It's also hard to implement request-response semantics using asynchronous messaging.</span></span> <span data-ttu-id="b724e-194">応答を送信するには、別のキューに加えて、要求メッセージと応答メッセージを関連付ける手段が必要です。</span><span class="sxs-lookup"><span data-stu-id="b724e-194">To send a response, you need another queue, plus a way to correlate request and response messages.</span></span>

## <a name="using-a-service-mesh"></a><span data-ttu-id="b724e-232">サービス メッシュの使用</span><span class="sxs-lookup"><span data-stu-id="b724e-232">Using a service mesh</span></span>
