## <a name="what-is-resiliency"></a><span data-ttu-id="f3555-116">回復性とは</span><span class="sxs-lookup"><span data-stu-id="f3555-116">What is resiliency?</span></span>
<span data-ttu-id="f3555-117">**回復性**とは、障害から回復して動作を続行する、システムの能力です。</span><span class="sxs-lookup"><span data-stu-id="f3555-117">**Resiliency** is the ability of a system to recover from failures and continue to function.</span></span> <span data-ttu-id="f3555-118">障害を*回避*することではなく、ダウンタイムまたはデータの損失を回避するように障害に*対応*することです。</span><span class="sxs-lookup"><span data-stu-id="f3555-118">It's not about *avoiding* failures, but *responding* to failures in a way that avoids downtime or data loss.</span></span> <span data-ttu-id="f3555-119">回復性の目的は、障害後にアプリケーションを完全に機能している状態に戻すことです。</span><span class="sxs-lookup"><span data-stu-id="f3555-119">The goal of resiliency is to return the application to a fully functioning state following a failure.</span></span>

<span data-ttu-id="f3555-120">回復性には、高可用性とディザスター リカバリーという 2 つの重要な側面があります。</span><span class="sxs-lookup"><span data-stu-id="f3555-120">Two important aspects of resiliency are high availability and disaster recovery.</span></span>

* <span data-ttu-id="f3555-121">**高可用性** (HA) は、大幅なダウンタイムなしに、正常な状態で実行を継続するアプリケーションの能力です。</span><span class="sxs-lookup"><span data-stu-id="f3555-121">**High availability** (HA) is the ability of the application to continue running in a healthy state, without significant downtime.</span></span> <span data-ttu-id="f3555-122">"正常な状態" とは、アプリケーションが応答し、ユーザーがアプリケーションに接続して対話できるという意味です。</span><span class="sxs-lookup"><span data-stu-id="f3555-122">By "healthy state," we mean the application is responsive, and users can connect to the application and interact with it.</span></span>  
* <span data-ttu-id="f3555-123">**ディザスター リカバリー** (DR) は、まれに発生する大きなインシデント、すなわち地域全体に影響するサービス中断のように、一時的なものではない大規模な障害から回復する能力です。</span><span class="sxs-lookup"><span data-stu-id="f3555-123">**Disaster recovery** (DR) is the ability to recover from rare but major incidents: non-transient, wide-scale failures, such as service disruption that affects an entire region.</span></span> <span data-ttu-id="f3555-124">ディザスター リカバリーには、データ バックアップやアーカイブの他に、バックアップからのデータベースの復元などの手動操作も含まれる場合があります。</span><span class="sxs-lookup"><span data-stu-id="f3555-124">Disaster recovery includes data backup and archiving, and may include manual intervention, such as restoring a database from backup.</span></span> 

<span data-ttu-id="f3555-125">HA と DR のとらえ方として、障害の影響が HA 設計で対応できる限度を超えたときに DR が始まると考えることができます。</span><span class="sxs-lookup"><span data-stu-id="f3555-125">One way to think about HA versus DR is that DR starts when the impact of a fault exceeds the ability of the HA design to handle it.</span></span> <span data-ttu-id="f3555-126">たとえば、ロード バランサーの背後に複数の VM を配置すると、いずれかの VM で障害が発生しても可用性は維持されますが、同時にすべての VM で障害が発生した場合は維持されません。</span><span class="sxs-lookup"><span data-stu-id="f3555-126">For example, putting several VMs behind a load balancer will provide availability if one VM fails, but not if they all fail at the same time.</span></span> 

<span data-ttu-id="f3555-127">回復力のあるアプリケーションを設計するときには、ご自分の可用性要件を理解する必要があります。</span><span class="sxs-lookup"><span data-stu-id="f3555-127">When you design an application to be resilient, you have to understand your availability requirements.</span></span> <span data-ttu-id="f3555-128">どの程度のダウンタイムを許容できますか。</span><span class="sxs-lookup"><span data-stu-id="f3555-128">How much downtime is acceptable?</span></span> <span data-ttu-id="f3555-129">これは、ある程度はコストによって決まります。</span><span class="sxs-lookup"><span data-stu-id="f3555-129">This is partly a function of cost.</span></span> <span data-ttu-id="f3555-130">起こり得るダウンタイムによってご自分のビジネスが被るコストはいくらになりますか。</span><span class="sxs-lookup"><span data-stu-id="f3555-130">How much will potential downtime cost your business?</span></span> <span data-ttu-id="f3555-131">アプリケーションの可用性を高めることにいくら投資したいですか。</span><span class="sxs-lookup"><span data-stu-id="f3555-131">How much should you invest in making the application highly available?</span></span> <span data-ttu-id="f3555-132">また、何をもってアプリケーションが使用可能であるとするかについても定義する必要があります。</span><span class="sxs-lookup"><span data-stu-id="f3555-132">You also have to define what it means for the application to be available.</span></span> <span data-ttu-id="f3555-133">たとえば、ユーザーが注文を送信しても、通常の時間枠内にシステムで処理できない場合、アプリケーションは "停止" 状態でしょうか。</span><span class="sxs-lookup"><span data-stu-id="f3555-133">For example, is the application "down" if a customer can submit an order but the system cannot process it within the normal timeframe?</span></span> <span data-ttu-id="f3555-134">また、特定の種類の障害が発生する可能性と、リスク軽減戦略のコスト効果が高いかどうかについても考慮します。</span><span class="sxs-lookup"><span data-stu-id="f3555-134">Also consider the probability of a particular type of outage occurring, and whether a mitigation strategy is cost-effective.</span></span>

<span data-ttu-id="f3555-135">もう 1 つの一般的な用語に**ビジネス継続性** (BC) があります。BC とは、自然災害やサービスの停止などの悪条件の発生時と発生後に、重要なビジネス機能を実行できることです。</span><span class="sxs-lookup"><span data-stu-id="f3555-135">Another common term is **business continuity** (BC), which is the ability to perform essential business functions during and after adverse conditions, such as a natural disaster or a downed service.</span></span> <span data-ttu-id="f3555-136">BC は、物理的な設備、人、コミュニティ、運輸、IT を含め、業務全体が対象です。</span><span class="sxs-lookup"><span data-stu-id="f3555-136">BC covers the entire operation of the business, including physical facilities, people, communications, transportation, and IT.</span></span> <span data-ttu-id="f3555-137">この記事ではクラウド アプリケーションを中心に扱いますが、回復性の計画は BC 要件全体のコンテキストで行う必要があります。</span><span class="sxs-lookup"><span data-stu-id="f3555-137">This article focuses on cloud applications, but resilience planning must be done in the context of overall BC requirements.</span></span> <span data-ttu-id="f3555-138">詳細については、National Institute of Science and Technology (NIST) の「[Contingency Planning Guide][capacity-planning-guide]」(危機管理計画ガイド) を参照してください。</span><span class="sxs-lookup"><span data-stu-id="f3555-138">For more information, see the [Contingency Planning Guide][capacity-planning-guide] from the National Institute of Science and Technology (NIST).</span></span>

## <a name="process-to-achieve-resiliency"></a><span data-ttu-id="f3555-139">回復性を実現するプロセス</span><span class="sxs-lookup"><span data-stu-id="f3555-139">Process to achieve resiliency</span></span>

## <a name="defining-your-resiliency-requirements"></a><span data-ttu-id="f3555-153">回復性の要件を定義する</span><span class="sxs-lookup"><span data-stu-id="f3555-153">Defining your resiliency requirements</span></span>
<span data-ttu-id="f3555-154">回復性の計画は、ビジネス要件から始まります。</span><span class="sxs-lookup"><span data-stu-id="f3555-154">Resiliency planning starts with business requirements.</span></span> <span data-ttu-id="f3555-155">ビジネス要件という点で回復性について考える場合の方法をいくつか挙げます。</span><span class="sxs-lookup"><span data-stu-id="f3555-155">Here are some approaches for thinking about resiliency in those terms.</span></span>

### <a name="decompose-by-workload"></a><span data-ttu-id="f3555-156">ワークロードごとに分解する</span><span class="sxs-lookup"><span data-stu-id="f3555-156">Decompose by workload</span></span>
<span data-ttu-id="f3555-157">多くのクラウド ソリューションは、複数のアプリケーション ワークロードで構成されます。</span><span class="sxs-lookup"><span data-stu-id="f3555-157">Many cloud solutions consist of multiple application workloads.</span></span> <span data-ttu-id="f3555-158">このコンテキストで "ワークロード" という用語は、ビジネス ロジックとデータ ストレージ要件の点で、他のタスクとは論理的に区別できる個別の機能またはコンピューティング タスクを意味します。</span><span class="sxs-lookup"><span data-stu-id="f3555-158">The term "workload" in this context means a discrete capability or computing task, which can be logically separated from other tasks, in terms of business logic and data storage requirements.</span></span> <span data-ttu-id="f3555-159">たとえば、eコマース アプリには次のようなワークロードがあります。</span><span class="sxs-lookup"><span data-stu-id="f3555-159">For example, an e-commerce app might include the following workloads:</span></span>

* <span data-ttu-id="f3555-160">製品カタログの閲覧と検索。</span><span class="sxs-lookup"><span data-stu-id="f3555-160">Browse and search a product catalog.</span></span>
* <span data-ttu-id="f3555-161">注文の作成と追跡。</span><span class="sxs-lookup"><span data-stu-id="f3555-161">Create and track orders.</span></span>
* <span data-ttu-id="f3555-162">推奨事項の表示。</span><span class="sxs-lookup"><span data-stu-id="f3555-162">View recommendations.</span></span>

<span data-ttu-id="f3555-163">これらのワークロードは、可用性、スケーラビリティ、データの一貫性、ディザスター リカバリーなどの要件が異なる可能性があります。</span><span class="sxs-lookup"><span data-stu-id="f3555-163">These workloads might have different requirements for availability, scalability, data consistency, disaster recovery, and so forth.</span></span> <span data-ttu-id="f3555-164">これらもビジネス上の決定事項です。</span><span class="sxs-lookup"><span data-stu-id="f3555-164">Again, these are business decisions.</span></span>

<span data-ttu-id="f3555-165">また、使用パターンについても考慮します。</span><span class="sxs-lookup"><span data-stu-id="f3555-165">Also consider usage patterns.</span></span> <span data-ttu-id="f3555-166">システムが使用可能である必要がある特定の重要な期間はありますか。</span><span class="sxs-lookup"><span data-stu-id="f3555-166">Are there certain critical periods when the system must be available?</span></span> <span data-ttu-id="f3555-167">たとえば、税金申告書サービスが申告期限の直前に停止してはならない、大規模なスポーツ イベント中にビデオ ストリーミング サービスは稼働している必要がある、などです。</span><span class="sxs-lookup"><span data-stu-id="f3555-167">For example, a tax-filing service can't go down right before the filing deadline, a video streaming service must stay up during a big sports event, and so on.</span></span> <span data-ttu-id="f3555-168">重要な期間中は、複数のリージョンにまたがる冗長的なデプロイにして、1 つのリージョンで障害が発生してもアプリケーションがフェールオーバーできるようにすることができます。</span><span class="sxs-lookup"><span data-stu-id="f3555-168">During the critical periods, you might have redundant deployments across several regions, so the application could fail over if one region failed.</span></span> <span data-ttu-id="f3555-169">ただし、複数リージョンのデプロイはコストが高くなるため、重要な期間以外は、1 つのリージョンでアプリケーションを実行することができます。</span><span class="sxs-lookup"><span data-stu-id="f3555-169">However, a multi-region deployment is more expensive, so during less critical times, you might run the application in a single region.</span></span>

### <a name="rto-and-rpo"></a><span data-ttu-id="f3555-170">RTO と RPO</span><span class="sxs-lookup"><span data-stu-id="f3555-170">RTO and RPO</span></span>
<span data-ttu-id="f3555-171">考慮すべき 2 つの重要なメトリックとして、目標復旧時間と回復ポイントの目標があります。</span><span class="sxs-lookup"><span data-stu-id="f3555-171">Two important metrics to consider are the recovery time objective and recovery point objective.</span></span>

* <span data-ttu-id="f3555-172">**目標復旧時間** (RTO) は、インシデントが発生した後に、アプリケーションに許容する使用不能状態の最大時間です。</span><span class="sxs-lookup"><span data-stu-id="f3555-172">**Recovery time objective** (RTO) is the maximum acceptable time that an application can be unavailable after an incident.</span></span> <span data-ttu-id="f3555-173">RTO が 90 分の場合、災害発生から 90 分以内にアプリケーションを実行状態に復元できる必要があります。</span><span class="sxs-lookup"><span data-stu-id="f3555-173">If your RTO is 90 minutes, you must be able to restore the application to a running state within 90 minutes from the start of a disaster.</span></span> <span data-ttu-id="f3555-174">RTO を非常に低い値にする場合、リージョン全体の停止から保護するために、スタンバイ状態で継続的に動作する第 2 のデプロイを維持する必要があります。</span><span class="sxs-lookup"><span data-stu-id="f3555-174">If you have a very low RTO, you might keep a second deployment continually running on standby, to protect against a regional outage.</span></span>
* <span data-ttu-id="f3555-175">**回復ポイントの目標** (RPO) は、災害発生時に許容できるデータ損失の最大期間です。</span><span class="sxs-lookup"><span data-stu-id="f3555-175">**Recovery point objective** (RPO) is the maximum duration of data loss that is acceptable during a disaster.</span></span> <span data-ttu-id="f3555-176">たとえば、単一のデータベースにデータを格納し、他のデータベースへのレプリケーションなしで、毎時のバックアップを実行する場合、最大 1 時間のデータを失う可能性があります。</span><span class="sxs-lookup"><span data-stu-id="f3555-176">For example, if you store data in a single database, with no replication to other databases, and perform hourly backups, you could lose up to an hour of data.</span></span>
